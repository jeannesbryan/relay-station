RELAY STATION: ARCHITECTURE & DEVELOPER GUIDE

CONFIDENTIAL: This document is the ultimate blueprint for future System Architects, Developers, or AI Assistants maintaining the Relay Station code. Read this before modifying the core engine.

1. THE CORE DOCTRINE (PHILOSOPHY)

Relay Station is not a standard web application; it is a decentralized, sovereign node. Every line of code must adhere to these four pillars:

  • Zero Dependency: Relay must run on the cheapest Shared Hosting possible. Do not introduce Composer, Node.js, or heavy third-party libraries. Stick to Vanilla PHP and Vanilla JS.
  • True P2P (No Center): Assume every other station can go offline, change domains, or turn hostile at any time. The node must survive in isolation.
  • Sovereignty over Consistency: If Node A disconnects from Node B, Node A immediately drops all incoming signals from Node B (including delete commands). Local sovereignty always overrides network consistency.
  • The SQLite Doctrine: Core memory relies entirely on SQLite in WAL (Write-Ahead Logging) mode with PRAGMA busy_timeout. This prevents "database locked" errors during simultaneous P2P signal bombardments.

2. SYSTEM ANATOMY (THE MAP)

If you want to build a new feature or fix a bug, here is where you need to look:

A. The Interface (UI/UX)

  • console.php: The Commander's Dashboard. Handles the main timeline, AJAX infinite scroll, Push-to-Talk (PTT) engine, WebP image compression, and station settings.
  • index.php: The Public Hologram. Displays visibility = 'public' transmissions to unauthenticated visitors. It strictly filters out foreign signals unless they are explicitly relayed (is_relay = 1).
  • direct.php: The Secure Channel. Handles End-to-End Encrypted (RSA-OAEP) direct messages.

B. The Core Engines (P2P Routing)

  • core/api_inbox.php: The Main Receiver (Shield). All incoming signals from the Constellation land here via POST requests. It handles Rate Limiting, Anti-Spoofing, The Anti-Loop Shield, and the Chain-Purge logic.
  • core/transmitter.php: The Main Launcher (Cannon). Handles local file uploads, saves the post to the local SQLite database, and fires the JSON payload to allied stations using curl_multi_exec for asynchronous broadcasting.

C. Network Protocols (Handshakes & Radar)

  • core/add_planet.php: Initiates a connection to a foreign node and fires the initial Handshake Token.
  • core/api_handshake.php: Receives incoming connection requests and enforces the Symmetric Key Exchange to prevent 401 Spoofing.
  • core/radar_sweep.php: Scans the Star Chart and automatically purges dead/offline nodes.

3. CORE MEMORY (DATABASE SCHEMA)

The database structure is defined in schema.sql.

  • transmissions: The main ledger for all broadcasts and direct messages.
    • is_remote: 0 (Local) or 1 (Incoming).
    • is_relay & origin_id: Used by The Relay Protocol. origin_id tracks the DNA of a message to prevent infinite Echo Chamber loops.
  • following / followers: The Star Chart. Stores URLs and the symmetric handshake_token.
  • signal_resonance: Stores 'Roger That' acknowledgments. Uses a UNIQUE constraint on (post_id, reactor_url) as a database-level Anti-Spam shield.
  • bookmarks: Uses INNER JOIN logic to save transmission IDs without duplicating message content.

4. ARCHITECT'S WORKFLOW GUIDE

"I want to build something new, where do I start?"

Scenario 1: Adding a New Type of Signal (e.g., a "Poll")

  1. UI (console.php): Add the button/form for the user to trigger the signal via AJAX.
  2. Sender (core/transmitter.php): Intercept the new signal type. Save it locally, assemble the JSON capsule, and route it to the target nodes.
  3. Receiver (core/api_inbox.php): Catch the incoming payload. Validate it, sanitize it, and INSERT it into the local database.

Scenario 2: Modifying Database Tables

  1. Never edit schema.sql for existing users: It is only read during a fresh installation.
  2. Use the Migration Script (core/upgrade_db.php): Write a safe try-catch block executing an ALTER TABLE command. The OTA updater will run this for all users.

Scenario 3: Modifying Outbound API Requests

You MUST include the WAF Bypass headers in any cURL request:

curl_setopt($ch, CURLOPT_HTTPHEADER, [
    'Content-Type: application/json',
    'Content-Length: ' . strlen($json_payload),
    'User-Agent: RelayStation-Transmitter/7.3'
]);

  • Cascading Chain-Purge: Forwarding destruction signals through trusted followers to ensure mirrored data is wiped across the entire Trust Chain.

6. LOCAL DEVELOPMENT & DEBUGGING

Developing a P2P system on a local machine requires special configurations since the node expects absolute URLs and SSL/HTTPS.

A. Simulating the Constellation

To test P2P features (like Relay or Purge) locally, you should run at least two instances of the station on different ports:

  • Node A: php -S localhost:8000
  • Node B: php -S localhost:8001

Note: You may need to temporarily disable or bypass the ssl_shield.php requirements in your development environment to allow non-HTTPS communication between local ports.

B. Tactical Debugging (Where to look for errors)

Since most P2P signals happen silently via cURL/AJAX, standard screen errors are rare. Use these logs instead:

  • Server-Side: Check the standard PHP error_log for cURL timeouts or SQLite PDO exceptions.
  • Client-Side: Use the Browser Developer Tools (F12) Console to monitor AJAX responses from console.php and api_inbox.php.
  • Database Integrity: Use an SQLite browser to inspect the data/relay_core.sqlite file directly and verify if origin_id or is_relay flags are correctly set.

C. Coding Conventions

  • No External Bloat: Always prefer built-in PHP functions over adding new files.
  • Security First: Every new entry point MUST include require_once 'core/ssl_shield.php' or equivalent protection.
  • WAF Compliance: Ensure all outbound requests mimic the RelayStation-Transmitter User-Agent to avoid being flagged as a bot.
